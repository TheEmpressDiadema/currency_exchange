REST API - стиль взаимодействия между клиентом и сервером через HTTP.
Принципы:
1. Клиент-серверная архитектура - клиент запрашивает данные, сервер предоставляет.
2. Отсутствие состояния(stateless) - Каждый запрос от клиента к серверу обрабатывается независимо от предыдущих запросов
Сервер не хранит информацию о состоянии клиента, что упрощает обработку и повышает отказоустойчивость системы
3. Кеширование - снижение нагрузки на сервер путем переиспользования повторных ответов(
    механизмы кеширования: cache-control, etag
)
4. единообразие интерфейса - все ресурсы REST API имеют схожую структуру и единообразные URL. Запросы выполняются с
помощью стандартных методов HTTP, данные передаются в JSON или XML
5. Наличие системы уровней - REST API может включать несколько независимых 
уровней(балансировщики нагрузки, прокси серверы, системы аутентификации)
6. REST API поддерживает выполнение и загрузку кода на стороне клиента

Компоненты REST API:
1. Ресурс - любая сущность, к которой можно получить доступ через API. Может быть пользователь, заказ, товар и др. объекты системы
Каждый ресурс имеет уникальный идентификатор

2. Методы HTTP - методы, которые используются для взаимодействия с ресурсами
2.1. GET - используется для запроса информации на ресурсе, не изменяет данные на сервере и многократный его вызов не должен 
приводить разные результаты
Пример запроса: GET /users/123 HTTP/1.1
Host: api.example.com
Пример ответа:
{
    "id":123,
    "name": "Иванов Иван Иванович"
}
код ответа: 200 OK, 404 Not found - если не найден
2.2. POST - Добавление нового ресурса, каждый вызов создает новый ресурс
Пример запроса: POST /users HTTP/1.1
HOST: api.example.com
Content-type: type
Вместе с запросом передается тело запроса:
{
    "name" : "someName"
}
Ответ сервера:
{
    "id": 124,
    "name" : "someName"
}
201 Created
2.3. PUT - Полное обновление существующего ресурса
Пример запроса:
PUT /users/123 HTTP/1.1
HOST: api.example.com
Content-Type: application/json
Тело Запроса:
{
    "name" : "someName"
}
ответ:
200 - если все ок
204 - no content
2.4. PATCH - Частичное обновление ресурса(изменяет только указанные поля)
Пример запроса:
PATCH /users/123 HTTP/1.1  
Host: api.example.com  
Content-Type: application/json
Тело запроса:
{
    "name" : "someName"
}
ответ сервера:
200 ОК, 204 - no content
2.5. DELETE - Удаление ресурса
пример запроса:
DELETE /users/123 HTTP/1.1
Host: api.example.com
Ответ сервера:
204 - no content если все ок, иначе 404 not found

3. Запросы и ответы: Обмен данными происходит через запрос к серверу и ответ от сервера в большинстве случаев используется JSON
